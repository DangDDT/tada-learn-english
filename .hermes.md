## SDLC Phases & Workflow

### Phase 1: Requirements & Planning
1. Create GitHub Issue (label: bug/enhancement/frontend/backend/ops)
2. Assign to Sprint Milestone
3. Spike if feasibility uncertain (skill: spike)
4. Write plan to .hermes/plans/ (skill: plan)

### Phase 2: Design & Architecture
1. Research via Context7 MCP
2. Write ADR to docs/architecture/
3. Update SDD if architecture changes
4. Sequential-thinking MCP for complex decisions

### Phase 3: Implementation
1. Branch: `git checkout -b feat/issue-N-description`
2. Delegate to OpenCode: `opencode run "task description"`
3. TDD cycle: RED -> GREEN -> REFACTOR
4. Commit with: `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`

### Phase 4: Testing
1. Unit tests (Go: go test, Frontend: jest)
2. Integration tests
3. E2E via Puppeteer MCP if needed

### Phase 5: Code Review & QA
1. Push branch, create PR
2. Self-review with requesting-code-review skill
3. GitHub MCP for PR review comments
4. Approve & merge

### Phase 6: CI/CD & Deploy
1. GitHub Actions workflows (.github/workflows/)
2. Docker Compose build & deploy
3. Health check verification

### Phase 7: Monitoring & Maintenance
1. Cron jobs for health checks
2. Session search for past context
3. Memory for persistent learnings

## Hard Gates (TD-011 Penalty)
1. **SOT-First:** No code without SRS/API-Spec/Use-Cases/SDD approval
2. **Bug-First:** ALWAYS log GitHub issue BEFORE fixing
3. **Branch Discipline:** Non-MVP work on separate branches, not main
4. Violation = Technical Debt entry in Backlog
